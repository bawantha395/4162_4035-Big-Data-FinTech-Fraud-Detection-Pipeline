# BankSecure Fraud Detection Pipeline - Project Report

## 1. Overview and goals
This project implements a real time fraud detection pipeline for a digital wallet provider. Transactions are generated by a mock producer, streamed through Kafka, analyzed by Spark Structured Streaming, persisted to PostgreSQL, and reconciled by a scheduled ETL job. The goals are to (1) detect fraud in real time using rule based logic, (2) keep a clean validated ledger for reporting and compliance, and (3) generate a repeatable analytic report that summarizes fraud attempts by merchant category. The pipeline is designed for local development using Docker, but the components and interfaces mirror production architecture patterns. The use case is deliberately simple, yet it exercises the core needs of a fintech workflow: low latency alerting, durable storage, periodic batch aggregation, and consistent reporting.

## 2. Tools and why they were selected
Kafka is used as the streaming backbone because it provides durable, ordered, and scalable ingestion. It decouples the producer from the detection layer and allows replay of events for testing. Spark Structured Streaming is chosen for processing because it supports event time windowing, watermarking, and stateful aggregations with clear semantics. It also integrates cleanly with Kafka, which makes the detection logic compact and transparent. PostgreSQL serves as the storage system for both all transactions and flagged transactions; its relational model is well suited for joins and ad hoc analysis needed by the reports. Airflow orchestrates the batch ETL because it provides time based scheduling, retry policies, and clear task ordering. Parquet is used as the warehouse format for validated transactions because it is compact, columnar, and common in analytical workflows. The tooling is simple but representative, and each component does one job well.

## 3. Data model and producer behavior
The mock producer generates transaction events that include account_id (user), event_ts (event time), merchant_category, amount, country (location), channel, currency, and flags when a transaction is fabricated as fraudulent. This aligns with the scenario requirement to produce user_id, timestamp, merchant_category, amount, and location, with periodic high value or impossible travel events. Fraud injection is controlled through a rate parameter and split into two rule types: high value transactions and impossible travel pairs where a second transaction appears in a different country within a short time window. This design gives a realistic mix of normal and anomalous activity, while keeping the fraud patterns deterministic enough to validate in tests and reports.

## 4. Streaming detection logic
The stream processor reads from Kafka, parses JSON into a structured schema, and then evaluates two rules. First, a high value rule flags any transaction where amount exceeds the threshold (default 5000). Second, an impossible travel rule groups by account_id and event time window, collects unique countries, and flags accounts with more than one country in the same ten minute window. The processor emits all transactions to the all_transactions table and alerts to the flagged_transactions table. This approach mirrors real world fraud systems that emit both normal and suspicious activity for audit and review. It is rule based rather than model based, which makes it explainable and easy to validate.

## 5. Event time vs processing time handling
Event time is captured in the transaction as event_ts, which is the timestamp produced by the simulator. In Spark, event_ts is parsed into a timestamp column called event_time and used in the windowed impossible travel rule. Processing time is tracked separately with proc_ts, which is set using current_timestamp at ingestion. This separation enables the pipeline to reason about late or out of order events. The stream uses a watermark of two minutes to allow small delays while bounding state growth. Windowing is defined on event_time, not processing time, so fraud detection reflects when the activity actually occurred rather than when it was processed. This is important in finance where the event timestamp represents customer behavior and should be used for rule evaluation.

## 6. Storage design and data flow
All ingested transactions are stored in PostgreSQL as all_transactions, and flagged alerts are stored in flagged_transactions. This dual storage makes it easy to run joins for analytics while keeping fraud alerts for immediate consumption. The ETL later writes validated records to a validated_ledger table, which is intended to represent the clean ledger for downstream reporting or compliance. The warehouse output is stored as Parquet files partitioned by run. This gives a durable snapshot of validated data and a path for offline analytics. The storage design balances speed and transparency: the streaming tables are fast for real time queries, and the warehouse files are efficient for batch analysis.

## 7. ETL pipeline and Airflow orchestration
Airflow schedules the ETL job every six hours. The task order is extract, transform, load, reconcile, and then generate the merchant report. Extraction pulls all transactions that are not present in flagged_transactions, which yields the clean set. Transformation aggregates totals by merchant and by country while also preserving clean rows for the ledger. Loading inserts rows into validated_ledger and writes a Parquet batch file for warehouse storage. Reconciliation compares total ingested amount to fraud amount and validated amount. This process is deterministic and repeatable, which is important for audits and testing. The same logic can be executed as a standalone script for local runs or unit testing.

## 8. Reconciliation logic
Reconciliation verifies that the financial totals align: expected_valid is computed as total_ingested minus fraud_amount, and then compared to the validated_ledger total. A small tolerance is allowed for rounding differences. If the absolute difference exceeds the tolerance, the status is FAIL. This is a simple but effective data quality gate. It surfaces pipeline issues such as missing inserts, duplicate rows, or an out of sync fraud table. In real deployments, a failed reconciliation would trigger an alert and block downstream reporting. The reconciliation log is stored in the database with a timestamp and totals, making it easy to track pipeline health over time.

## 9. Analytic reporting
Two reports are generated. The merchant report focuses on fraud attempts by merchant category and is written as a CSV in the deliverables folder. The full analytical report is a text file that includes transaction volume, fraud by type, fraud by merchant and country, top flagged accounts, and the latest reconciliation status. These reports serve two purposes: they provide evidence that the pipeline is working end to end, and they produce a human readable artifact for grading. The report format is plain text and CSV so that it can be inspected without specialized tools and easily attached to submissions.

## 10. Deployment notes and configuration
The pipeline is designed to run locally using Docker Compose for Kafka, Zookeeper, and PostgreSQL. Spark runs in local mode with defined driver and executor memory. A checkpoint directory is required for streaming state, and it is configured to use a local file path to avoid HDFS dependencies. Environment variables are used for connection details, thresholds, and rates. A run script starts the services, creates topics, launches the detector, runs the producer, and then executes ETL and reports. This makes the system reproducible and easy to demo. The run script can be adapted for longer durations or different fraud rates.

## 11. Limitations and future work
The current system uses static rules and does not include model based scoring or feedback loops. It also assumes a single Kafka topic and does not support schema evolution. The impossible travel rule uses a fixed window and may miss patterns that span longer intervals. There is no deduplication beyond simple primary key constraints, which may be insufficient for real world retries. The ETL assumes that fraud alerts are final, while in real systems alerts may be reviewed and reclassified. Future work could include alert review workflows, supervised models, user level risk scoring, and integration with a dashboard for investigations. It could also include better observability, such as metrics and structured logs.

## 12. Ethics and data governance
Fraud detection inherently involves sensitive personal and financial data. Even in a mock pipeline, it is important to model governance and privacy practices. First, data minimization should apply: only fields required for detection and reconciliation should be retained. Location data should be coarse enough to support rules without enabling unnecessary tracking. Second, access controls are essential: only authorized roles should access flagged alerts or detailed transaction history. Audit logs should record access to sensitive data. Third, retention limits should be enforced so that historical data is removed or anonymized after its operational need expires. Fourth, explainability matters because fraud decisions can impact customers. Rule based alerts should be documented and human review should exist before punitive action. Finally, bias and fairness should be monitored. Even simple rules can disproportionately impact certain regions or user groups if the data is skewed. Governance policies should define review processes, allow appeals, and require periodic audits of detection logic. In production, compliance with regulations such as PCI DSS, GDPR, and local banking laws would be mandatory.

## 13. Conclusion
The BankSecure pipeline satisfies the scenario requirements by combining a streaming fraud detector with a scheduled batch ETL and analytic reporting. It uses proven tools, clear data flow, and explicit event time handling. The result is a working prototype that demonstrates real time alerts, validated ledger creation, and reconciliation. The deliverables include architecture documentation, source code, and generated reports that show the pipeline in action. This design is simple enough to explain and evaluate, yet faithful to the kinds of systems used in financial fraud monitoring.
